## What I found

- The app reads the backend credentials from `.env` using:
  - `VITE_SUPABASE_URL`
  - `VITE_SUPABASE_PUBLISHABLE_KEY`
  - `VITE_SUPABASE_PROJECT_ID`
- Your local `.env` format should be correct if it contains those exact names.
- The published/anon key is the value of `VITE_SUPABASE_PUBLISHABLE_KEY` in the `.env` file generated by Lovable Cloud. In Lovable, you can double-check it under **Cloud → Secrets** or from the project’s generated `.env` if you downloaded/exported it.
- The hosted backend currently reports healthy, but direct database reads are still timing out intermittently. That explains why local data and login can fail even with a correct `.env`: the local app still depends on the same online backend.

## Immediate fix I recommend

Build a presentation-safe fallback mode so the app works locally even when the school/client network blocks or slows the backend.

### 1. Keep the live backend for normal use
- Leave the existing Lovable Cloud connection as the default.
- Published app and preview keep using real database/auth/admin features.

### 2. Add local demo fallback for public data
- If backend data requests fail or time out, automatically load the bundled seed data from `src/data/seedData.ts`.
- This keeps the homepage, search, discover, dashboard, gene pages, and disease pages usable during the presentation.
- Show a subtle banner like: “Demo data shown while reconnecting.”

### 3. Add query retry/cache settings
- Configure React Query with short retries and longer cache time.
- This prevents one dropped request from making everything appear empty.

### 4. Add a local demo login bypass only for local development
- For `localhost` only, allow a demo admin login if the real login fails.
- This would unlock the admin interface for presentation/demo purposes without affecting the published website.
- The published site will still require real authentication.

### 5. Add clear local setup instructions
- Add a short `LOCAL_DEMO.md` with:
  - required `.env` names
  - where to find the publishable key
  - `npm install` / `npm run dev`
  - how to use local demo fallback if the backend is unreachable

## Local `.env` checklist

Use this exact shape:

```text
VITE_SUPABASE_PROJECT_ID="oawsxaoniadbdwvtxtdm"
VITE_SUPABASE_URL="https://oawsxaoniadbdwvtxtdm.supabase.co"
VITE_SUPABASE_PUBLISHABLE_KEY="your-publishable-key-here"
```

Important:
- The key must be named `VITE_SUPABASE_PUBLISHABLE_KEY`, not `SUPABASE_ANON_KEY`.
- After changing `.env`, fully stop and restart `npm run dev`.
- Do not open the app from `file://`; use the Vite URL, usually `http://localhost:5173`.

## User roles/capabilities summary

- **Public visitor**: can browse/search/dashboard/discover/detail pages and submit suggestions.
- **Authenticated basic user**: same as public unless assigned a role.
- **Manager**: sees Admin link, can review suggestions, edit genes/diseases/associations/categories, and sees edit buttons on detail pages.
- **Super Admin**: everything Manager can do, plus User Management/whitelist/temp key controls.
